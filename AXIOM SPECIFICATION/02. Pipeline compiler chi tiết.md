Dưới đây là thiết kế chi tiết cho **Axiom Compiler Pipeline** (Trình biên dịch `axc`). Kế thừa tốc độ biên dịch siêu tốc của V-lang, khả năng thực thi lúc biên dịch của Jai, phân tích lười biếng của Zig, và tối ưu hóa bộ nhớ qua Đồ thị kết nối (Connection Graph), pipeline này đảm bảo thời gian build dưới 1 giây (sub-second) trong khi vẫn xuất ra mã tối ưu hóa sâu.

---

### 1. Lexer (Trình phân tích từ vựng)
Biến chuỗi ký tự thô thành các thẻ từ (tokens).

*   **Input:** Source code (`.ax` file, UTF-8 string).
*   **Output:** Mảng phẳng các Tokens (Flat Token Array).
*   **Data Structure:** Mảng 1 chiều không cấp phát chuỗi (Zero-copy). Mỗi Token chỉ chiếm 8 bytes: `struct { kind: u8, offset: u32, length: u32 }`. Không lưu chuỗi thực tế mà chỉ lưu con trỏ trỏ về source.
*   **Thuật toán:** Automata hữu hạn tất định (DFA) quét tuần tự.
*   **Complexity:** $O(N)$ với $N$ là độ dài source code.
*   **Optimization Opportunities:**
    *   **SIMD Acceleration:** Sử dụng AVX2/NEON để quét qua các khoảng trắng (whitespace), thụt lề (indentation) và comment với tốc độ 16-32 bytes/chu kỳ CPU.
    *   **Zero-allocation:** Hoàn toàn không gọi `malloc` hay tạo object `String` mới trên Heap.

### 2. Parser (Trình phân tích cú pháp)
Chuyển đổi luồng token thành cấu trúc ngữ pháp có thứ bậc.

*   **Input:** Flat Token Array.
*   **Output:** Cây cú pháp trừu tượng dạng mảng phẳng (Flat AST).
*   **Data Structure:** `Array<AstNode>`. Thay vì dùng con trỏ (tốn 8 bytes và gây trượt cache), node sử dụng chỉ mục (Index) 32-bit: `struct { kind: u8, child_idx: u32, sibling_idx: u32, token_idx: u32 }`.
*   **Thuật toán:** Đệ quy xuống (Recursive Descent Parsing), Top-Down, LL(1) thân thiện. Axiom không có tiền xử lý macro C (No C-preprocessor) nên ngữ pháp 100% Context-Free.
*   **Complexity:** $O(T)$ với $T$ là số lượng tokens.
*   **Optimization Opportunities:**
    *   **Lock-free Parallel Parsing:** Vì không có file header (như `#include` của C++), mỗi file `.ax` được đưa cho một luồng (thread) riêng biệt để parse độc lập hoàn toàn mà không cần khóa (mutex).

### 3. AST $\rightarrow$ Semantic Graph (Đồ thị ngữ nghĩa & Phân tích lười biếng)
Giải quyết quan hệ không gian tên (namespaces), imports, và nạp chồng (overloads).

*   **Input:** Tập hợp các Flat AST từ nhiều file.
*   **Output:** Semantic Graph (Đồ thị các Symbol đã được resolve).
*   **Data Structure:** Symbol Table dưới dạng Hash Map mở (Open-addressing Hash Map) tĩnh.
*   **Thuật toán:** **Lazy Field Analysis** (học từ Zig). Trình biên dịch không phân tích toàn bộ code. Nó bắt đầu từ `main()`, duyệt theo đồ thị phụ thuộc. Nếu một hàm hoặc một trường (field) của struct không bao giờ được gọi tới, nó bị bỏ qua hoàn toàn.
*   **Complexity:** $O(U \cdot \log S)$ với $U$ là số symbol thực sự được dùng, $S$ là tổng symbol.
*   **Optimization Opportunities:**
    *   **Incremental Hash Caching:** Hash AST của các module không đổi và cache lại. Sửa file nào chỉ rebuild nhánh đồ thị của file đó.

### 4. Type Checker & Connection Graph (Hệ thống kiểu & Phân tích bộ nhớ)
Kiểm tra an toàn kiểu tĩnh và theo dõi vòng đời đối tượng.

*   **Input:** Semantic Graph.
*   **Output:** Typed AST + Escape Map (Bản đồ thoát bộ nhớ).
*   **Data Structure:** **Connection Graph** (Đồ thị kết nối). Node là các Object / Reference, Edge là sự trỏ đến. Đồ thị này theo dõi trạng thái phân nhánh luồng điều khiển (path-sensitive).
*   **Thuật toán:**
    1.  **Hindley-Milner Local Inference:** Suy luận kiểu từ dưới lên (Bottom-up).
    2.  **Monomorphization:** Nhân bản các hàm Generics cho từng kiểu dữ liệu cụ thể.
    3.  **Compile-time Execution (`#run`):** Nếu gặp chỉ thị chạy lúc biên dịch, đưa đoạn AST vào máy ảo (Bytecode JIT) thực thi ngay lập tức, trả về hằng số tĩnh và gắn lại vào Typed AST.
    4.  **Escape Analysis:** Duyệt Connection Graph. Nếu object không thoát khỏi hàm hiện tại, đánh dấu `StackAlloc`.
*   **Complexity:** $O(V + E)$ trên Đồ thị kết nối cục bộ của mỗi hàm.
*   **Optimization Opportunities:**
    *   Tự động chèn các hook `=destroy` và `free` ở cuối block, loại bỏ nhu cầu sử dụng GC lúc runtime.

### 5. Intermediate Representation (Axiom IR - AIR)
Hạ cấp Typed AST thành biểu diễn tuyến tính dễ tối ưu.

*   **Input:** Typed AST.
*   **Output:** AIR (Mạng lưới Basic Blocks dạng SSA - Static Single Assignment).
*   **Data Structure:** Mảng 3 địa chỉ (3-address code instructions): `opcode, type, dest, arg1, arg2`.
*   **Thuật toán:** Post-order traversal của Typed AST.
*   **Complexity:** $O(N)$ (N = số lượng node của AST).
*   **Optimization Opportunities:**
    *   Chèn siêu dữ liệu AI (AI Metadata) vào IR để chuẩn bị xuất ra file `.axmeta` cho các mô hình ngôn ngữ lớn (LLM).

### 6. Optimization (Tối ưu hóa)
Biến đổi luồng dữ liệu để phần mềm chạy với hiệu suất cao nhất.

*   **Input:** AIR thuần.
*   **Output:** Optimized AIR.
*   **Data Structure:** Control Flow Graph (CFG) và Data Flow Graph.
*   **Thuật toán:**
    1.  **Constant Folding & Propagation:** Gộp hằng số.
    2.  **Dead Code Elimination:** Xóa khối lệnh không với tới.
    3.  **Data-Oriented Mutation:** Dựa vào annotation `@SOA` hoặc AI hints, xoay trục bộ nhớ từ `Array of Structures` (AoS) thành `Structure of Arrays` (SoA) trên các vòng lặp đồ họa/game.
    4.  **Compile-Time Garbage Collection (CTGC):** Thay vì `malloc` mới, sử dụng Đồ thị kết nối để tìm object vừa chết và gán địa chỉ đó cho object mới chuẩn bị cấp phát (In-place memory reuse).
*   **Complexity:** Tùy thuộc vào số lượng Passes, thường giới hạn ở $O(I \log I)$ ($I$ = instructions).
*   **Optimization Opportunities:** Giới hạn chỉ chạy ở cờ `-O3` (Release). Ở chế độ Debug (`-O0`), bỏ qua toàn bộ bước này để build < 0.1s.

### 7. Codegen (Trình sinh mã)
Chuyển đổi AIR tối ưu thành mã thực thi. Axiom sử dụng chiến lược **Dual Backend**.

*   **Input:** Optimized AIR.
*   **Output:** C Source Code (cho MVP) hoặc Machine Code (x86-64/ARM).
*   **Data Structure:** String Builder (C-Backend) hoặc Mảng Bytes (Native).
*   **Thuật toán:**
    *   *Trường hợp 1 (C-Backend):* Ánh xạ các lệnh SSA sang các cấu trúc tương ứng của chuẩn C11/C23 (ví dụ dùng macro `defer` mới của C23 hoặc các `#pragma simd`).
    *   *Trường hợp 2 (Native):* Linear Scan Register Allocation (Thuật toán quét tuyến tính) để gán biến ảo vào các thanh ghi vật lý (RAX, RBX, v.v.).
*   **Complexity:** $O(I)$ tuyến tính, cực kỳ nhanh.
*   **Optimization Opportunities:** C-Backend là một vũ khí cực mạnh ở MVP. Nó cho phép Axiom tận dụng hàng thập kỷ tối ưu hóa của GCC/Clang, đồng thời chạy được trên mọi OS/Vi điều khiển lập tức.

### 8. Object File & Linker (Liên kết)
Đóng gói mã máy thành file nhị phân.

*   **Input:** C Code / Machine Code.
*   **Output:** File thực thi cuối cùng (Executable - `.exe`, ELF, Mach-O).
*   **Thuật toán & Data Structure:**
    *   *C-Backend:* Gọi tiến trình phụ (subprocess) `gcc/clang -O3 -flto` ngầm định ở nền để tự động link.
    *   *Native Backend:* Viết trực tiếp mã máy vào bộ nhớ (In-memory linker). Resolving các symbols (Jump offsets) trên RAM và xả (dump) thẳng thành file chuẩn ELF/PE mà không cần tạo các file `.o` trung gian ghi xuống ổ cứng.
*   **Complexity:** Phụ thuộc vào kích thước mảng byte.
*   **Optimization Opportunities:**
    *   **Nhúng AI Metadata:** Đính kèm toàn bộ Đồ thị ngữ nghĩa tĩnh dạng JSON (đã nén) vào phân vùng độc lập `.axmeta` ngay trong file nhị phân. File thực thi của Axiom lúc này không chỉ chạy được trên CPU, mà còn là một kho tàng tri thức (Knowledge Graph) để các AI Agent chẩn đoán, debug ngược lại logic gốc mà không cần code nguồn.
