# AXIOM LANGUAGE SPECIFICATION v1.0 (Part 10: AI Semantic Layer)

## 47. TỔNG QUAN KIẾN TRÚC LỚP NGỮ NGHĨA

### 47.1. Rationale (Cơ sở lý luận)
Khi AI (như GitHub Copilot) đọc mã Rust hay Python, nó chỉ nhìn thấy "văn bản phẳng" (flat text) hoặc cùng lắm là Cây cú pháp trừu tượng (AST) thông qua Tree-sitter. Nó không hiểu thực sự biến nào đang sở hữu vùng nhớ nào, hay hàm nào có hiệu ứng phụ (side-effects). 

Kế thừa ý tưởng tích hợp phân tích ngữ nghĩa sâu của dự án Carbon và Đồ thị Kết nối (Connection Graph), Axiom xuất toàn bộ "nhận thức" của Trình biên dịch về chương trình thành một Đồ thị Tri thức Thực thi (Executable Knowledge Graph). AI sẽ truy vấn đồ thị này để hiểu code thay vì chỉ đọc chuỗi văn bản.

## 48. SEMANTIC SCHEMA (LƯỢC ĐỒ NGỮ NGHĨA)

### 48.1. Rationale
Lược đồ ngữ nghĩa (Semantic Schema) định nghĩa cấu trúc dữ liệu mô tả mối quan hệ không gian và thời gian của mọi thực thể trong chương trình. Nó giải quyết triệt để sự mơ hồ (ambiguity) để AI có thể thực hiện tái cấu trúc mã (refactoring) một cách tất định.

### 48.2. Cấu trúc Đồ thị (Graph Structure)
Lược đồ sử dụng khái niệm **Connection Graph** (Đồ thị kết nối) kết hợp với Cây Cú pháp Phẳng (Flat AST).
*   **Nodes (Đỉnh):**
    *   `ValueNode`: Đại diện cho một giá trị trên RAM (Stack hoặc Heap).
    *   `RefNode`: Đại diện cho con trỏ Tham chiếu thế hệ (Generational Reference).
    *   `OperationNode`: Đại diện cho một phép toán hoặc lời gọi hàm (`CallExpr`).
*   **Edges (Cạnh):**
    *   `Owns`: Quan hệ sở hữu duy nhất (Single Ownership).
    *   `Borrows`: Quan hệ mượn (View).
    *   `FlowsTo`: Luồng dữ liệu (Def-Use Chain).
    *   `EscapesTo`: Cạnh đặc biệt đánh dấu một biến nội bộ đã "thoát" (escape) ra khỏi hàm.

### 48.3. Constraints & Trade-offs
*   *Constraint:* AI bị nghiêm cấm thay đổi cấu trúc `Owns` của Đồ thị nếu không có sự cho phép rõ ràng từ lập trình viên, nhằm đảm bảo tính an toàn bộ nhớ.
*   *Trade-off:* Việc theo dõi mọi cạnh `FlowsTo` làm tăng kích thước bộ nhớ khi biên dịch (Compile-time memory overhead). Do đó, Axiom chỉ xuất đồ thị này dưới dạng siêu dữ liệu khi có cờ biên dịch `--emit-meta`.

## 49. METADATA FORMAT (`.axmeta`)

### 49.1. Rationale
Định dạng `.axmeta` là cầu nối giao tiếp nhị phân/văn bản giữa Axiom Compiler và AI Agents. Nó được đóng gói trực tiếp vào file nhị phân tĩnh hoặc xuất ra dưới dạng tệp phụ (sidecar file).

### 49.2. Định dạng & Schema
Tệp sử dụng chuẩn JSON (để AI dễ parse) hoặc MessagePack (để tối ưu I/O).

**Ví dụ JSON payload (.axmeta):**
```json
{
  "version": "1.0",
  "module": "std.net.http",
  "symbols": {
    "req": {
      "type": "HttpRequest",
      "memory_class": "Heap",
      "generation_id": "tracked",
      "capabilities": ["Isolated"]
    }
  },
  "connection_graph": {
    "nodes": [{"id": 1, "type": "RefNode", "name": "req"}],
    "edges": [{"from": 1, "to": 2, "relation": "EscapesTo", "target": "Thread_2"}]
  },
  "effects_profile": {
    "fetch_data": ["IOError", "NetworkLatency"]
  }
}
```

### 49.3. Ứng dụng thực tế
Khi người dùng hỏi AI: *"Tại sao biến `req` ở đây lại bị lỗi?"*. AI sẽ không đoán bừa. Nó truy vấn khối `"edges"` trong `.axmeta`, phát hiện `req` có quan hệ `EscapesTo` sang luồng khác nhưng thiếu khóa năng lực (capability) `Isolated`, từ đó đưa ra câu trả lời chính xác 100% về mặt ngữ nghĩa.

## 50. AST ANNOTATIONS (AI REASONING HOOKS)

### 50.1. Rationale
Lập trình viên cần một ngôn ngữ chung để giao tiếp với AI (Intent-driven programming). Bằng cách sử dụng các chỉ thị (Annotations / Pragmas) gắn thẳng vào AST, lập trình viên "neo" (hook) các quy tắc suy luận mà AI bắt buộc phải tuân theo khi sinh mã.

### 50.2. Examples (Các điểm móc suy luận)
*   **`@[ai::assert_pure]`**: Đánh dấu một khối mã. Nếu AI Copilot tự động sinh code (codegen) vào đây, Trình biên dịch Axiom sẽ kiểm tra chéo (Cross-check) và từ chối ngay lập tức nếu code sinh ra chứa I/O hoặc thay đổi trạng thái toàn cục.
*   **`@[ai::suggest(layout="SoA")]`**: Hướng dẫn AI Agent phân tích và đề xuất việc xoay trục bộ nhớ (Structure of Arrays) cho struct nhằm tối ưu hóa CPU Cache.
*   **`@[ai::explain(depth="low-level")]`**: Trình biên dịch tự động gọi LLM cục bộ (Local LLM) để sinh ra Docstring giải thích cơ chế quản lý bộ nhớ của hàm này, thay vì lập trình viên phải tự viết tài liệu.

### 50.3. Constraints
*   Các chỉ thị `@[ai::...]` hoàn toàn không có tác động đến mã máy sinh ra (Zero runtime overhead). Chúng chỉ có ý nghĩa đối với bộ tiền xử lý Linter và AI LSP.

## 51. AI OPTIMIZATION APIs (`std.compiler.ai`)

### 51.1. Rationale
Tận dụng sức mạnh thao tác AST thời gian biên dịch (AST Macros) tương tự Nim, và Jai, Axiom cung cấp bộ API để một chương trình AI bên ngoài có thể nhúng tay trực tiếp vào quá trình tối ưu hóa mã (Optimization pipeline).

### 51.2. Cơ chế Hoạt động (Semantic API)
Trình biên dịch `axc` mở một kênh RPC cục bộ khi dùng cờ `--ai-server`.
AI Agent có thể gửi các lệnh sửa đổi AST vào kênh này.

**API Schema:**
```axiom
import std.compiler.ast
import std.compiler.ai

// AI Agent gọi API này tại thời gian biên dịch (Compile-time)
#run fn ai_refactor_loop(node_id: u32) -> Result[AstNode, AiError]:
    let graph = ai.get_semantic_graph(node_id)
    
    // AI tự tính toán và đề xuất cấu trúc vòng lặp SIMD/Vectorized
    let optimized_ast = ai.propose_vectorization(graph)
    
    // Trình biên dịch PHẢI xác minh tính hợp lệ (Memory & Type Safety)
    if compiler.verify_semantics(optimized_ast):
        return Ok(optimized_ast)
    else:
        return Err("Hallucination: AI violated Ownership rules")
```

### 51.3. Đánh đổi (Trade-offs)
*   *Độ trễ biên dịch (Compile-time latency):* Việc để AI can thiệp vào AST có thể đẩy thời gian biên dịch từ $0.1$ giây lên vài chục giây (phụ thuộc vào tốc độ phản hồi của LLM). Do đó, API này chỉ được kích hoạt trong các đợt "Deep Optimization Pass" (Tối ưu hóa sâu) trước khi release, không dùng cho quá trình gõ code hàng ngày.

## 52. SEMANTIC DEBUGGING (GỠ LỖI NGỮ NGHĨA)

### 52.1. Rationale
Gỡ lỗi truyền thống (GDB/LLDB) dựa trên địa chỉ bộ nhớ và thanh ghi vật lý. Semantic Debugging nâng cấp trải nghiệm này: Trình gỡ lỗi ánh xạ ngược (map back) các lỗi runtime trực tiếp lên Đồ thị Tri thức (Knowledge Graph), cho phép AI giải thích lỗi bằng ngôn ngữ tự nhiên cấp cao.

### 52.2. Cơ chế (Semantic Trace)
*   Khi chương trình Axiom gặp lỗi (ví dụ: `Use-After-Free` bị chặn bởi Generational References), Runtime không in ra dải Hexa khó hiểu. Nó xuất ra file `.axtrace`.
*   `.axtrace` chứa bản sao trạng thái của Đồ thị Kết nối (Connection Graph) ngay tại thời điểm sập.
*   **Trải nghiệm người dùng:** Lập trình viên mở `.axtrace` bằng AI LSP. AI sẽ phân tích và nói:
    > *"Lỗi giải tham chiếu thế hệ (Generational ID Mismatch) tại dòng 42. Nguyên nhân ngữ nghĩa: Biến `Session` đã bị hàm `=destroy` thu hồi tại dòng 30 (Hết vòng đời), nhưng Actor số 5 vẫn cố gắng đọc View (Lent reference) của nó qua channel."*

### 52.3. Constraints
*   Để Semantic Debugging hoạt động, mã nhị phân phải được biên dịch ở chế độ Debug (`-O0`) hoặc kèm theo cờ `--emit-meta`, nhằm giữ lại toàn bộ Bảng Ký hiệu (Symbol Table) và Dấu vết Đồ thị (Graph Trace). Việc này sẽ làm tăng kích thước tệp thực thi.
