### 1. SO SÁNH CÁC KỸ THUẬT PARSING

| Kỹ thuật | Cơ chế hoạt động | Ưu điểm | Nhược điểm | Độ phù hợp với Axiom |
| :--- | :--- | :--- | :--- | :--- |
| **Recursive Descent (Đệ quy xuống)** | Viết tay các hàm đệ quy tương ứng với từng luật ngữ pháp (Top-down). | Cực kỳ nhanh, dễ gỡ lỗi, dễ dàng chèn logic xử lý thụt lề (indentation) qua cấu trúc ngăn xếp. Cung cấp Error Recovery tốt nhất. | Phải viết tay lượng code lớn. Dễ bị lặp vô hạn nếu ngữ pháp có đệ quy trái (Left-recursion). | **Rất cao.** Được sử dụng bởi GCC, Clang, V, và Nim. |
| **Pratt Parser (Top-Down Operator Precedence)** | Gắn "trọng số ưu tiên" (precedence) cho mỗi toán tử để quyết định hướng phân tích. | Xử lý các biểu thức toán học (Expressions) cực kỳ thanh lịch và tuyến tính mà không cần tạo ra hàng chục hàm đệ quy sâu. | Chỉ phù hợp để phân tích biểu thức (Expressions), không dùng để phân tích cấu trúc khối (Statements). | **Rất cao (kết hợp).** |
| **PEG (Parsing Expression Grammar)** | Ngữ pháp sử dụng toán tử "Lựa chọn có thứ tự" (Ordered choice - `/`),. | Đảm bảo tính tất định (100% Deterministic) do không có tính mơ hồ. Có thể dùng tool sinh code. | Có thể tiêu tốn bộ nhớ khổng lồ nếu dùng kỹ thuật Packrat (memoization) để đạt tốc độ $O(N)$. | **Trung bình.** Ý tưởng tốt nhưng overhead bộ nhớ lớn. |
| **LR / LALR** | Sinh tự động bằng công cụ như Yacc/Bison (Bottom-up). | Xử lý ngữ pháp phức tạp tốt, cảnh báo sớm về tính mơ hồ (Shift/Reduce conflicts). | Rất khó xử lý cú pháp dựa trên thụt lề (indentation-based). Error Recovery cực kỳ tệ (thường chỉ văng ra `Syntax Error`). | **Thấp.** Khó sinh ra Flat AST như thiết kế tùy chỉnh của Axiom. |
| **GLR (Generalized LR)** | Thử tất cả các nhánh có thể nếu gặp cú pháp mơ hồ (Ambiguity). | Phân tích được mọi loại ngữ pháp context-free. Dùng trong C++ parser. | Phân tích chậm, phi tất định (Non-deterministic), tốn nhiều nhánh bộ nhớ. | **Rất thấp.** Đi ngược lại triết lý Deterministic của Axiom. |

#### **LỰA CHỌN CHO AXIOM:** 
Kiến trúc lý tưởng nhất cho Axiom là sự kết hợp: **Hand-written Recursive Descent (cho khối lệnh / Statements)** + **Pratt Parser (cho biểu thức / Expressions)**. 

Lý do:
1. Axiom sử dụng cú pháp thụt lề (Indentation-based). Recursive Descent cho phép chúng ta dễ dàng duy trì một ngăn xếp trạng thái thụt lề (Indentation Stack) ngay trong lúc parse.
2. Trình phân tích đệ quy viết tay (Hand-written) kiểm soát hoàn hảo việc cấp phát bộ nhớ mảng phẳng (Flat AST), cho phép nhúng trực tiếp Metadata cho AI.

### 2. PARSER ARCHITECTURE (KIẾN TRÚC PARSER)

Trình phân tích của Axiom không cấp phát bộ nhớ động rải rác. Nó duyệt qua mảng Tokens (được Lexer tạo ra dưới dạng Zero-copy) và ghi liên tục vào một mảng cấu trúc AST.

**Luồng dữ liệu (Dataflow):**
1. **Lexing State:** Quét mảng Token. Khi gặp token thụt lề (indent), đẩy mức độ khoảng trắng (space count) vào `indent_stack`.
2. **Statement Parsing (Recursive Descent):** Các hàm như `parse_if_stmt()`, `parse_fn_decl()` tiêu thụ token từ trái qua phải.
3. **Expression Parsing (Pratt):** Hàm `parse_expression(precedence)` dùng bảng trọng số tĩnh để nhóm các toán tử (+, *, <=) thành các Node tính toán với chi phí đệ quy bằng 0.

### 3. AST SCHEMA (LƯỢC ĐỒ CÂY CÚ PHÁP PHẲNG)

Để tối ưu hóa CPU Cache và tránh hiện tượng phân mảnh, AST của Axiom là một mảng liền kề (Array of Structs / SoA). Không có bất kỳ con trỏ (Pointer) nào được sử dụng. Mối quan hệ Cha-Con được quản lý bằng **Chỉ mục 32-bit (32-bit Indices)**.

```rust
// Mã giả minh họa cấu trúc của một AST Node trong Axiom Compiler
struct AstNode {
    kind: NodeKind,           // 8-bit Enum (e.g., IfStmt, BinOp, VarDecl)
    token_idx: u32,           // 32-bit trỏ về vị trí Token gốc để báo lỗi / trích xuất tên
    
    // Mối quan hệ cấu trúc cây bằng Index (Tiết kiệm bộ nhớ so với con trỏ 64-bit)
    first_child: u32,         // Index của node con đầu tiên (0 nếu không có)
    next_sibling: u32,        // Index của node anh em tiếp theo cùng cấp (0 nếu không có)
    
    // Dữ liệu phụ thuộc vào loại Node
    payload: u32,             // Ví dụ: Lưu TypeID (sau Semantic check) hoặc Precedence
}

// Bố cục mảng toàn cục
struct Ast {
    nodes: Array<AstNode>,
    // Rễ của AST module
    root_idx: u32,
}
```

*Rationale:* Khi AI hoặc Compiler đọc AST, nó có thể duyệt tuyến tính (Linear Scan) qua mảng `nodes` để đếm số lượng biến hoặc tìm các hàm, đạt băng thông bộ nhớ bộ nhớ tối đa thay vì nhảy con trỏ ngẫu nhiên qua Heap.

### 4. SYNTAX ERROR RECOVERY (PHỤC HỒI LỖI CÚ PHÁP)

Mục tiêu là trình biên dịch phải tìm ra **nhiều lỗi nhất có thể** trong một lần chạy thay vì crash ngay ở lỗi đầu tiên. Axiom sử dụng chiến lược **Panic Mode with Synchronization Tokens**.

**Thuật toán Phục hồi:**
1. Khi phát hiện cú pháp sai (ví dụ: thiếu kiểu dữ liệu trong khai báo `mut x := `), Parser đánh dấu Node hiện tại là `ErrorNode` và ghi lỗi vào mảng Log.
2. Parser kích hoạt **Panic Mode**.
3. **Skip Tokens (Bỏ qua Token):** Parser sẽ nuốt (consume) các token liên tiếp cho đến khi nó bắt gặp một **Synchronization Token** (Token đồng bộ).
    * *Cấp biểu thức:* Đồng bộ tại các token: `)`, `]`, `,`, `NEWLINE`.
    * *Cấp khối lệnh:* Đồng bộ tại các mức thụt lề bằng hoặc thấp hơn cấp hiện tại (dựa vào `indent_stack`), hoặc các từ khóa ranh giới như `fn`, `struct`, `if`.
4. Sau khi đồng bộ thành công, Parser thoát Panic Mode và tiếp tục phân tích đoạn mã tiếp theo. Mảng AST vẫn giữ nguyên cấu trúc phẳng với các nhánh hỏng (ErrorNodes) bị cô lập tĩnh.

### 5. INCREMENTAL PARSING (PHÂN TÍCH TĂNG DẦN)

Biên dịch tăng dần cực hạn (Extreme Incremental Compilation) là yếu tố giúp thời gian build của Axiom gần bằng 0 khi lập trình viên gõ code. 

**Kiến trúc xử lý:**
* Thay vì phân tích lại toàn bộ file nguồn dài 10,000 dòng, Parser chia file thành các **Top-level Blocks** (Khối cấp cao nhất, thường bắt đầu với thụt lề cấp 0).
* Axiom Compiler duy trì một bảng Băm (Hash Table) nội tại nối `Function_Name -> Hash(AST_Subtree)`.
* **Thuật toán:**
  1. Khi người dùng sửa file, Lexer chỉ quét (rescan) lại khu vực văn bản bị thay đổi.
  2. Bằng cách so sánh độ sâu thụt lề (Indentation level), Parser xác định chính xác ranh giới của hàm/khối đang bị chỉnh sửa (ví dụ: chỉ sửa bên trong `fn calculate()`).
  3. Parser chỉ chạy lại thuật toán Recursive Descent cho riêng block đó.
  4. Nạp đè (Patch) nhánh AST con mới vào mảng `AstNode` phẳng (các Node cũ được gắn cờ `DeadNode` để tái sử dụng). Đồ thị ngữ nghĩa tĩnh (Connection Graph) ở các phần khác của chương trình không bị ảnh hưởng.

Cách tiếp cận này không chỉ giúp biên dịch lại hàm đang sửa (Hot-reloading) trong vài mili-giây, mà còn cho phép AI Agent (qua `.axmeta`) ngay lập tức nhận thức được sự thay đổi của cấu trúc mà không phải chịu độ trễ phân giải toàn bộ hệ thống.
